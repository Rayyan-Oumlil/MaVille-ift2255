1. Cadre du projet

1.1 Description du projet

Contexte

À Montréal, la présence fréquente de travaux publics et privés fait partie du quotidien des citoyens. Ces interventions, parfois mal planifiées ou insuffisamment signalées, engendrent régulièrement des désagréments : ralentissements de la circulation, fermetures temporaires de rues, obstacles pour les piétons… Ces perturbations contribuent à un climat d’incertitude et d’insatisfaction parmi les résidents, qui voient leur mobilité et leur routine souvent bouleversées sans préavis.

Problèmes à résoudre

  • Communication déficiente : Les résidents sont souvent surpris par des interventions touchant leur mobilité
  • Processus manuel : Le STPM utilise actuellement des fichiers Excel et des communications manuelles
  • Manque de coordination : Les échanges entre résidents, prestataires et STPM sont inefficaces
  • Absence de notifications : Pas de système automatisé pour informer les parties prenantes

Proposition

Le STPM propose la création de l'application MaVille pour automatiser et améliorer la communication des activités et travaux. Cette application permettra une collaboration plus étroite entre les résidents, les prestataires et le STPM, réduisant significativement les incompréhensions entre les différents acteurs.

1.2 Hypothèses

Hypothèse 1 : Portée du prototype

Ce premier devoir se concentre sur la modélisation et un prototype fonctionnel de base. Les aspects de conception détaillée et d'implémentation complète seront abordés dans les devoirs 2 et 3 respectivement.

Hypothèse 2 : Interface simplifiée

Pour ce prototype initial, nous utilisons une interface en ligne de commande qui permet de valider la logique métier sans se préoccuper de l'interface graphique finale.

Hypothèse 3 : Stockage temporaire

Les données sont stockées en mémoire pour ce prototype. L'intégration d'une base de données persistante sera considérée lors de la phase d'implémentation (Devoir 3).

Hypothèse 4 : Authentification reportée

L'authentification et la gestion des sessions utilisateurs sont volontairement simplifiées dans ce prototype pour se concentrer sur les fonctionnalités métier essentielles.

Hypothèse 5 : Validation de concept

Ce prototype vise principalement à démontrer la faisabilité des cas d'utilisation principaux et à valider notre compréhension du domaine métier avant la conception détaillée.

2. Modélisation du domaine

2.1 Glossaire

  • STPM : Service des Travaux Publics de Montréal, responsable de la gestion des signalements et des projets de travaux.
  • Résident : Citoyen de Montréal pouvant signaler un problème sur la voie publique.
  • Prestataire : Entreprise ou organisme proposant des projets pour résoudre des problèmes signalés.
  • Problème : Incident de voirie signalé par un résident (ex. : nid-de-poule, fissure, etc.).
  • Signalement : Action de signaler un problème au STPM, accompagnée d'une description et de la localisation.
  • Fiche problème : Document interne contenant les détails d'un signalement (type, localisation, priorité).
  • Priorité : Niveau d'urgence du problème : faible, moyenne ou élevée.
  • Candidature : Proposition d'un prestataire pour prendre en charge un ou plusieurs problèmes.
  • Projet : Ensemble de travaux validés à réaliser, incluant dates, budget et prestataire attitré.
  • Notification : Message envoyé automatiquement à un utilisateur concernant un changement de statut d'un projet ou d'un problème.
  • StatutProjet : État d'un projet (ex. : EN_ATTENTE, EN_COURS, TERMINE).
  • StatutCandidature : État d'une candidature (EN_ATTENTE, APPROUVEE, REJETEE).

4. Prototype

4.1 Architecture du prototype

Le prototype suit une architecture en couches :

  • Couche UI (ca.udem.maville.ui) : Gestion de l'interface utilisateur console
  • Couche Service (ca.udem.maville.service) : Logique métier et gestionnaires
  • Couche Modèle (ca.udem.maville.modele) : Entités du domaine

4.2 Fonctionnalités implémentées

--> Fonctionnalités pour les résidents

  • Signalement de problèmes avec toutes les informations requises
  • Consultation de ses signalements par email
  • Consultation des travaux en cours par arrondissement

--> Fonctionnalités pour les prestataires

  • Consultation libre des problèmes en attente
  • Soumission de candidatures avec formulaire complet
  • Gestion des candidatures (consulter, modifier, annuler)

--> Fonctionnalités techniques

  • Menu principal avec choix de profil
  • Validation des saisies utilisateur
  • Gestion des erreurs et messages informatifs
  • Code modulaire et commenté

4.3 Instructions d'utilisation

Exécution via le JAR (recommandé)

 1. Téléchargez le fichier maVille-1.0-SNAPSHOT.jar depuis la section "Releases" du projet GitHub.
 2. Ouvrez un terminal et placez-vous dans le dossier où se trouve le fichier JAR.
 3. Exécutez la commande suivante :
java -jar maVille-1.0-SNAPSHOT.jar

Alternative : Utilisation avec un IDE (IntelliJ, VSCODE, etc.)

  1. Importez le projet en tant que projet Maven dans votre IDE préféré.
  2. Lancez la classe ca.udem.maville.Main (clic droit > "Run" ou équivalent).

Navigation dans l'application

  1. Au démarrage, choisir le profil (Résident ou Prestataire)
  2. Naviguer dans les menus avec les numéros
  3. Utiliser "0" pour revenir au menu précédent
  4. Suivre les instructions à l'écran pour les saisies
  5. Choisir "Quitter" dans le menu principal pour terminer

4.4 Limitations et améliorations futures

Limitations actuelles

  • Données non persistantes (perdues à la fermeture)
  • Interface console uniquement
  • Authentification simplifiée
  • Pas de notifications automatiques

Améliorations futures

  • Interface graphique web ou mobile
  • Base de données persistante
  • Système d'authentification complet
  • Intégration avec les systèmes municipaux existants

Conclusion

Ce prototype démontre la faisabilité de l'application MaVille et implémente les fonctionnalités essentielles demandées. Il constitue une base solide pour le développement d'une solution complète qui automatisera et améliorera significativement la gestion des travaux publics à Montréal.
